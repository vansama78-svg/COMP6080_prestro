# Accessibility (A11Y)

This document summarises accessibility-related choices in the Presto frontend.

## Semantic structure and landmarks

- Main navigation flows use descriptive headings (`h1` on dashboard and editor).
- The slide toolbox is grouped with `role="group"` and `aria-label="Slide tools"` so assistive technologies can identify the control cluster.
- Modal dialogs use `role="dialog"`, `aria-modal="true"`, and titled via `h2` / `aria-labelledby` where appropriate (e.g. slide control panel).

## Forms and labels

- Form controls are associated with visible `<label>` elements using `htmlFor` / `id`.
- Colour inputs use labels (“Text colour”, gradient “From” / “To”) so colour is not the only cue when adjusting backgrounds.

## Keyboard support

- Left / right arrow keys navigate between slides in the editor and in preview when focus is not inside an input, textarea, or select.
- Arrow navigation is suppressed while modals are open so keyboard events do not move slides unintentionally behind a dialog.

## Images and embedded content

- User-supplied images require alt text in the editor modal; thumbnails and slide images render with meaningful `alt` where the user provided it.
- YouTube embeds use `iframe` elements with a `title` derived from the element id for screen reader context.

## Focus and visibility

- Disabled navigation buttons remain in the tab order but are visibly distinct (`opacity` / cursor) per existing deck styling.
- Slide control panel lists slides as an unordered list with buttons for predictable structure.

## Future improvements

- Trap focus inside large modals and return focus to the triggering control on close.
- Announce slide changes via a live region for users who rely on screen readers during slideshow navigation.
